# ADR-009: Wix Data Schema for Cases, Workflow Templates, Organizations, and Tasks

**Status:** Accepted
**Date:** 2026-07-21

## Context

Through Phase 13, Beacon has organization-scoped domain architecture (Phase 11's workflow templates, Phase 13's authentication/authorization) but zero real backend storage — every `services/*` function reads `services/__mocks__/fixtures.ts`. Before any real Wix Data integration can begin, the minimum schema this domain model actually needs has to be designed: what collections exist, what fields they carry, how tenant isolation is enforced at the data layer (not just in application code), and how the two immutability guarantees the codebase already promises — `Case.intakeOwnerId` never changing after creation, and a `Case.workflowSnapshot` never depending on later `WorkflowTemplate` edits — translate into a real, queryable schema rather than an in-memory array's happenstance behavior.

`docs/CMS_SCHEMA.md`, the pre-existing schema document, predates both Phase 11 and Phase 13: it has no `workflowTemplateId`/`workflowTemplateVersion`/`workflowSnapshot` fields on `Cases`, and no membership/authenticated-identity model at all. It also documents a `decedentFirstName`/`decedentLastName` split that was never actually implemented (the real `Case` type has always used one `decedentName` field) — it was speculative from the start, not a contract this phase needs to honor field-for-field.

## Decision

Six collections: `organizations`, `organizationMemberships`, `workflowTemplates`, `workflowTemplateVersions`, `cases`, `tasks` — see `docs/WIX_DATA_SCHEMA.md` for the complete field-level specification. Four evaluated candidates (`users`/`userProfiles`, `caseTimelineEvents`, `caseDocuments` metadata, `auditEvents`) are explicitly not created, each for a documented reason rather than by omission.

Three shape decisions carry the most weight:

1. **Wix system `_id` is never used as a Beacon domain identifier.** Every collection gets its own Beacon-generated `beacon<Thing>Id` text field, and every cross-collection reference is a plain text field holding another collection's `beacon<Thing>Id` — not a formal Wix "Reference" field type. This keeps Beacon's authorization and service code entirely ignorant of Wix's internal bookkeeping, so a future data export/reimport (which can regenerate internal ids) can never silently break domain identity or a case's own stable id.
2. **Template identity is split from version identity into two collections** (`workflowTemplates`/`workflowTemplateVersions`), rather than the in-memory `WorkflowTemplate.versions: WorkflowTemplateVersion[]` array `types/workflowTemplate.ts` currently uses. An in-memory array's "immutability" is just a convention no one violates yet; a real database needs the append-only guarantee to be a property of a *collection* (only ever inserted into, never updated) so that a `Case`'s embedded `workflowSnapshot` plus its recorded `workflowTemplateVersion` number remain resolvable and trustworthy indefinitely, independent of whatever the template's *current* latest version is.
3. **`intakeOwnerId`/`caseHandlerId`/`assigneeId` are specified against the authenticated-identity space** (`organizationMemberships.userId`), not the pre-existing `StaffProfile.id` space the application currently, actually uses. This is a deliberate design-ahead-of-code decision: today's `hooks/useSession.ts` and `services/casesService.ts` still derive case ownership from a hardcoded `StaffProfile` stub with no connection to Phase 13's real login. The schema is built for where the domain model should end up — case ownership tied to who actually authenticated — and that gap is recorded as a known limitation, not silently resolved by changing application code this phase (out of scope: "schema-only").

Tenant isolation is enforced the same way Phase 13 already enforces organization authorization: every organization-owned collection carries `organizationId`, every Wix collection defaults to backend/API-Key-only permissions (no public or member-self read), and the real guarantee is that Beacon's own server code always derives `organizationId` from `resolveAuthorizationContext()` before issuing any query — Wix permissions are a defensive backstop, never the sole mechanism.

## Consequences

- `docs/WIX_DATA_SCHEMA.md` becomes the authoritative schema reference for these six collections; `docs/CMS_SCHEMA.md`'s `Cases`/`CaseTasks`/`StaffProfiles` sections are superseded (not deleted — `CaseContacts`/`CaseLogEntries` remain out of this phase's scope and stay documented there).
- No collection has actually been created in Wix as of this ADR. Creating them requires either working Wix MCP tool access (undiscoverable in every session to date) or manual creation by a human in the Wix dashboard, following `docs/WIX_DATA_SCHEMA.md` exactly.
- Two immutability guarantees (`workflowTemplateVersions`' append-only behavior, `workflowTemplates.organizationId`'s conditional requirement) are application-enforced, not database-enforced — Wix Data has no native support for either. This is a real, recorded gap: a future service bug could still call `.update()` against a version row Wix Data itself would not reject.
- The `StaffProfile` vs. authenticated-identity fork remains open. Nothing about today's running application changes — `useSession()` still returns the same hardcoded stub it always has — but the schema is no longer aligned with what that stub produces. Resolving this (retiring `StaffProfile` in favor of `organizationMemberships`, and rewiring `casesService.create`) is recommended Phase 15/16 work.
- No mock data was migrated, and no `services/*` function was changed to read or write Wix. `DATA_ADAPTER=mock` remains the default and the only functioning mode.

## Alternatives Considered

- **Use a formal Wix Data "Reference" field type for cross-collection links** (e.g., `cases.organizationId` as a true Reference to `organizations`, keyed by Wix's internal `_id`): rejected — it would make Beacon's own domain identifiers (`Case.id`, `WorkflowTemplate.id`, etc.) dependent on Wix's internal record ids, the opposite of "separate Wix record metadata from Beacon domain identifiers." Plain text fields holding another collection's own `beacon<Thing>Id` keep the two entirely decoupled.
- **Keep `WorkflowTemplateVersion` as a nested array field on `workflowTemplates`**, matching today's in-memory `WorkflowTemplate.versions` shape exactly: rejected — a nested array field in Wix Data has no way to be independently indexed, queried by version, or protected from an accidental in-place edit the way a dedicated append-only collection can be disciplined to behave. Splitting the two collections is what makes "immutable historical versions" an enforceable property instead of a hopeful convention.
- **Reference `StaffProfile.id` for `intakeOwnerId`/`caseHandlerId`, matching what the application code does today**: rejected — Phase 13's own stated intent (`intakeOwnerId` "derived automatically from the trusted session") only makes sense if it's actually the trusted session's identity, not a separate, unauthenticated staff directory. Building the schema around today's actual (arguably already-outdated) code would bake in a gap Phase 13 was supposed to close.
- **Create a `staffProfiles` Wix collection alongside `organizationMemberships`**: rejected as unnecessary duplication of the same underlying concept ("which humans work at this organization") — recorded as a recommendation to unify on `organizationMemberships` instead, not implemented this phase.
- **Add `caseTimelineEvents`/`auditEvents` collections now, anticipating future need**: rejected — both would be built for a documented future possibility with no current consumer (the activity log is fully derived at read time; no audit-logging code exists anywhere in the app), directly against this phase's own "do not create unnecessary collections merely for future possibilities" instruction.
