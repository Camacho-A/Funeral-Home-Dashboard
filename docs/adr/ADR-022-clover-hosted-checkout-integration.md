# ADR-022: Clover Hosted Checkout Integration

**Status:** Accepted
**Date:** 2026-07-24

## Context

ADR-021 (Secure Payment Architecture) eliminated card-data storage and built the client-side isolation and server-side guard that make it structurally impossible for Beacon to receive a PAN, CVV, or expiration date â€” but it explicitly stopped short of integrating a real payment provider ("Do not implement Stripe, Square, or any payment provider yet" was that phase's own instruction). This phase (19B) is that integration: Manor's Cremation's Clover merchant account, via Clover's **Hosted Checkout** product, wired into Beacon's existing case workflow.

## Why Hosted Checkout was selected

Clover offers more than one integration surface. Hosted Checkout was chosen â€” and the phase's own instructions explicitly ruled out Clover's legacy Developer Pay API â€” because it keeps Beacon's PCI scope as small as possible: the customer enters card data on a page Clover serves and controls entirely; Beacon's server only ever creates a checkout session (a POST with an amount and a redirect URL) and later receives a signed webhook telling it what happened. No card data ever transits Beacon's frontend, Beacon's server, or any Beacon-controlled network hop â€” the alternative (an embedded iframe or client-side tokenizing SDK) would still keep Beacon out of PAN-handling scope but adds a real frontend integration surface this phase didn't need to build to satisfy "launch Clover directly from a case, complete a payment on Clover's hosted page, return to Beacon."

## Payment sequence

See the [published sequence diagram](https://claude.ai/code/artifact/01709693-f31d-4be5-a1fa-d2c17d555188) produced before implementation. Summary: staff clicks "Collect with Clover" on `PaymentCard` â†’ Beacon creates a pending `PaymentRecord` and a Clover Hosted Checkout session â†’ the browser does a full redirect to Clover's hosted URL â†’ the customer pays â†’ Clover redirects back to Beacon's `/cases/[caseId]/payments/return` page **and, independently, sends a signed webhook to `/api/webhooks/clover`** â†’ the webhook is Beacon's authoritative confirmation, updating the `PaymentRecord` and the case; the return page shows "Confirming paymentâ€¦" and polls until it observes that same, webhook-confirmed status.

## Browser redirect versus webhook authority

**The return-page redirect is never treated as proof of payment.** Clover's own documentation only specifies the redirect as a UX convenience ("set redirect URLs for customers... in case of successful, failed, or incomplete transactions") â€” it carries no signature, no server-verifiable claim, and query parameters on a browser redirect are trivially forgeable by anyone who knows (or guesses) a `paymentId`. `app/(portal)/cases/[caseId]/payments/return/page.tsx` reflects this directly in its own code: arriving with `?outcome=success` does *nothing* by itself â€” the page shows "Confirming paymentâ€¦" and polls `GET /api/cases/[caseId]/payments/[paymentId]` (via `useCasePaymentStatus`, a 2.5s interval that stops once a terminal status is reached) until that endpoint reports a status that was set by the webhook (or, as a fallback, the same endpoint's own best-effort server-to-server reconciliation with Clover â€” never by the redirect).

The one narrow exception: a `?outcome=cancel` redirect is allowed to mark the `PaymentRecord` `cancelled` directly (`app/api/cases/[caseId]/payments/[paymentId]/cancel/route.ts`), because "cancelled" makes no claim that money changed hands â€” the worst case of a wrongly-applied cancel is an available retry, never a false "paid" state.

## Secret management

Clover credentials are never committed and never stored in Wix. `types/payment.ts`'s `PaymentIntegration` stores only *names* of environment variables (`merchantIdReference`, `credentialReference`, `webhookSecretReference`) â€” `lib/clover/cloverConfig.ts` is the only code that resolves those names into actual values, read from `process.env` at call time, never logged, never returned in any API response. This indirection is what lets a future per-organization encrypted-secret store replace env-var resolution with something more sophisticated without changing `PaymentIntegration`'s shape or any caller. (`merchantId` was originally stored as a plain, non-secret Wix field; a correction pass folded it into the same env-var-reference pattern as the other two, purely for one consistent resolution mechanism â€” not because it's actually secret.)

For Manor's Cremation (the initial tenant), a `paymentIntegrations` row exists in the live Wix site with `environment: "sandbox"`, `isEnabled: false`, and explicitly sandbox-labeled reference names (`CLOVER_MANORS_SANDBOX_MERCHANT_ID`, `CLOVER_MANORS_SANDBOX_PRIVATE_KEY`, `CLOVER_MANORS_SANDBOX_WEBHOOK_SECRET`) â€” no real sandbox credentials were available during this phase, so nothing is connected yet. A future production row must name a **separate** set of variables (e.g. `CLOVER_MANORS_PRODUCTION_*`) â€” never these â€” so activating production can never accidentally run against sandbox credentials or vice versa. See "Required production activation steps" in the phase report.

## Organization isolation

Every checkout-creation request re-derives its organization from the caller's session via `requireAuthorizedOrganization` (the same helper every other Route Handler uses since Phase 15X) â€” never trusts a client-supplied `organizationId`. The webhook path has no session at all (Clover is the caller), so isolation works differently there: the incoming event's `data` field (Clover's checkout session id) is used to look up the corresponding `PaymentRecord` â€” which already carries the correct `organizationId`, recorded when Beacon created the session â€” and only *then* is that organization's own `PaymentIntegration.webhookSecretReference` used to verify the signature. A verified signature alone only proves the event came from *some* Clover merchant holding that secret; the webhook route additionally confirms the event's own `merchantId` matches the resolved organization's configured merchant id (`getCloverMerchantId(integration)`) before applying any update â€” closing the gap a shared or misconfigured secret could otherwise open between organizations.

## Idempotency (correction pass)

The original design used an application-level scan (`findReusablePendingRecord`: look for an existing pending record matching case/amount/currency/purpose within Clover's 15-minute session lifetime) to avoid creating a duplicate Clover session on a double-click or reload. **This was insufficient under concurrency** â€” two near-simultaneous requests could both complete the scan before either had inserted its record, both concluding "no existing session," and both proceeding to create a Clover session. This was corrected to a real, atomic, database-enforced guard:

- `PaymentRecord` gained an `idempotencyKey` field. The client (`PaymentCard`) generates a fresh UUID once per logical "attempt" (held in component state, regenerated only once the attempt settles â€” see that component's own comment) â€” a genuine double-click or an in-flight retry of the same click carries the *identical* key; a deliberately new click after the prior one has settled gets a fresh one.
- **Wix Data's unique index supports only a single field, not a genuine multi-field compound constraint** (confirmed from Wix's own API schema: `unique` indexes are capped at 1 field, vs. 3 for a regular index) â€” so the literal "unique compound index on organizationId + idempotencyKey" isn't directly buildable. The equivalent guarantee is achieved by composing the *value* instead: the stored `idempotencyKey` is `{organizationId}:{clientKey}`, and a single-field unique index (`unique_idempotencyKey`) is built on that composed value â€” two different organizations can never collide (the value is prefixed with the organization id itself, not merely improbable to collide), and two requests for the same organization+key collide exactly as a true compound unique index would have.
- `services/paymentsService.ts`'s `createIdempotentPendingPaymentRecord` attempts the insert directly; Wix returns HTTP 409 (confirmed empirically against the live `paymentRecords` collection, error code `WDE0123`) on a duplicate value, including for two rows sharing an *empty string* â€” the same as any other value. The loser of the race catches this 409 and looks up the winner's already-inserted record (`findPaymentRecordByIdempotencyKey`) and returns that instead of erroring or creating a duplicate. Because Wix enforces this at the database level, not via a prior read in Beacon's own process, this is correct even under genuine concurrency and even across multiple server instances.
- **Empty-value collision, confirmed and designed around:** the original design also inserted a new pending record with `providerCheckoutId: ''` (Clover hasn't assigned a real session id yet at that point) and separately unique-indexed `providerCheckoutId`. Empirically, two rows with `providerCheckoutId: ''` collide under Wix's unique index exactly like any other duplicate value (`WDE0123`) â€” meaning a second pending payment could never have been created while a first was still awaiting its Clover session. Corrected two ways: (1) `providerCheckoutId` is now seeded with a placeholder unique to its own record (`pending:{id}`), never an empty string; (2) the unique index that used to live on `providerCheckoutId` was dropped (Wix caps a collection at exactly **one** unique index total) and rebuilt on `idempotencyKey` instead, which is what actually needed the atomic guarantee â€” `providerCheckoutId` still has a regular (non-unique) compound index (`organizationId, providerCheckoutId`) for the webhook-adjacent update path, sufficient since correctness there comes from the idempotencyKey guard upstream, not from a `providerCheckoutId` uniqueness constraint.

On the webhook side, a second, independent layer remains: `PaymentRecord.status` only ever transitions away from `pending` once â€” a stale, genuinely-different event (e.g. a late DECLINED) arriving after a terminal one is a no-op (`app/api/webhooks/clover/route.ts`'s own explicit `if (record.status !== 'pending') return`). See "Durable webhook deduplication" below for *repeat delivery* of the *same* event, which this status check alone does not durably guarantee.

## Durable webhook deduplication (two correction passes)

**First pass:** relying solely on the `status !== 'pending'` check above for *duplicate delivery* (as opposed to *out-of-order, different* events) was flagged as insufficient: it conflates "this payment has reached a terminal state" with "this specific webhook delivery was already processed," and doesn't durably prove correctness is guaranteed by a real atomic constraint. Clover's Hosted Checkout webhook payload has no distinct, stable event id separate from the payment's own id, and Clover's documentation doesn't specify delivery guarantees (at-least-once vs. exactly-once, retry behavior). Beacon computes its own deterministic `eventFingerprint` â€” `sha256(merchantId|checkoutSessionId|paymentId|status)` â€” from the already-signature-verified payload's stable fields, deliberately **excluding** the delivery timestamp (a genuine retry of the same event carries a different `Clover-Signature` `t=` value, but must still fingerprint identically).

**Second pass (final):** the first pass's design â€” insert the fingerprint as the `webhookEvents` item's `_id`, treat a 409 as "duplicate, skip" â€” had its own real gap: it conflated *claiming* an event with *successfully finishing* it. If the fingerprint row was inserted but the downstream `PaymentRecord` update then failed (a genuine, if rare, Wix write failure, or the process being killed mid-request), the fingerprint would be permanently "claimed" â€” any later retry from Clover would hit the 409 branch and be silently acknowledged (200) without the update ever actually being applied. Corrected to a full lifecycle, tracked per `webhookEvents` row (`types/webhookEvent.ts`'s `WebhookEventRecord`):

- **States:** `processing` â†’ `completed` (success) or `failed` (the update didn't take).
- **Claim** (`services/paymentsService.ts`'s `claimWebhookEvent`): attempts to insert a fresh `processing` row keyed by the fingerprint. A brand-new fingerprint's claim is fully atomic â€” Wix's `_id` uniqueness enforces it, not a prior read. On a 409, the existing row is fetched and reconciled: `completed` â†’ acknowledge without reapplying; `processing` and still fresh â†’ acknowledge without applying concurrently (another live request is handling it); `failed`, or `processing` but stale (past `STALE_PROCESSING_TIMEOUT_MS`, 2 minutes â€” generous relative to how long processing actually takes) â†’ **reclaim**: transition back to `processing`, increment `attemptCount`, proceed to reprocess.
- **Complete** (`markWebhookEventCompleted`) is called **only** after the `PaymentRecord` update has actually succeeded (and, if newly `succeeded`, after `markCasePaidIfVerified` has run) â€” this is what makes "claimed" and "done" genuinely distinct states.
- **Fail** (`markWebhookEventFailed`) is called whenever processing doesn't complete â€” the route also returns a non-2xx status in that case, so Clover retries; the retry's identical fingerprint reclaims the `failed` row rather than being told "already handled."
- **Reclaim is best-effort, not a true compare-and-swap** â€” Wix Data has no CAS primitive, so reclaiming a `failed`/stale-`processing` row is a plain read-then-update, which could in principle race with another reclaim attempt. This is an accepted, documented trade-off: the actual money-moving safety net is `PaymentRecord.status`'s own transition-only-from-`pending` guard (unchanged, still fully idempotent), so a reclaim race can at worst cause a harmless duplicate reprocessing attempt, never an incorrect final payment state. Only the *initial* claim of a brand-new fingerprint â€” the common case â€” needed, and has, true atomicity.
- **A claim-time database failure is never acknowledged as success** â€” `claimWebhookEvent` throws on any insert failure that isn't a genuine duplicate-key 409, and the route's own try/catch turns that into a 503, never `{received: true}` â€” verified with a dedicated wix-mode test that forces `insertWixDataItem` to reject with a plain (non-`WixDataApiError`) error.
- **`attemptCount`, `firstReceivedAt`, `lastAttemptAt`, `completedAt`** are persisted throughout for reconciliation â€” `firstReceivedAt` never changes across a reclaim; the others update on every attempt/completion.

Mock mode's equivalent (`services/__mocks__/paymentFixtures.ts`'s `webhookEventFixtures`, a plain `Map`) is explicitly **not** durable â€” consistent with every other mock-mode data store in this codebase, which never claims durability. The durability guarantee this correction calls for is a wix-mode property, backed by a real Wix collection, not a mock-mode one.

## Multiple-payment support

A `PaymentRecord` is a row per payment *attempt*, never a singleton on `Case` â€” `paymentRecords (organizationId, caseId)` is a one-to-many relationship by design, so a case can accumulate a failed attempt, a successful retry, a deposit, and a later balance payment, all independently visible in `PaymentCard`'s history list. Nothing in the schema or the checkout route prevents starting a new payment regardless of existing history.

## PCI boundary

Beacon's server and Wix Data collections never receive or store a PAN, CVV, or expiration date at any point in this flow â€” the checkout-creation request body Beacon sends to Clover contains only an amount, a line-item description, and Beacon's own non-sensitive correlation metadata (a `note` field embedding the Beacon payment id and case id â€” the one place Clover's checkout-session body accepts free-form metadata); the webhook Beacon receives back contains only a status, Clover's own payment/checkout identifiers, and (via a supplementary standard Clover Payments API call, once a `providerPaymentId` is known) a card brand and last four digits. This keeps Beacon itself out of PCI DSS scope for cardholder data storage â€” the actual PAN never exists in any system Beacon controls.

## Migration path to Clover iframe

If a more integrated (non-redirect) checkout experience is wanted later, Clover's iframe-based Ecommerce SDK is the natural next step â€” it would replace the full-page redirect in `PaymentCard`'s "Collect with Clover" handler and the dedicated `/payments/return` page with an in-page card-collection iframe (still Clover-hosted, still outside Beacon's own DOM/JS reach â€” an iframe boundary, not a form Beacon renders), while `lib/paymentProvider.ts`'s interface, `PaymentRecord`/`PaymentIntegration`'s schema, the webhook route's signature verification, and the idempotency/multi-payment model would all remain unchanged. The provider-neutral `PaymentProvider` interface this phase introduces is exactly the seam such a change â€” or a future Stripe/Square provider entirely â€” would be built behind.

## Consequences

- Manor's Cremation staff can collect a real, verified payment against any case, with a visible history of every attempt.
- No raw card data can reach Beacon at any layer â€” proved by the test suite (`lib/paymentFieldGuard.ts` reuse, explicit "no card fields in requests/responses/fixtures" assertions) and by design (Clover's hosted page is the only place a card is ever typed).
- A future Stripe/Square integration is additive: a new `lib/<provider>/*Provider.ts` implementing `PaymentProvider`, no changes to case-domain logic, routes' shapes, or the `PaymentRecord`/`PaymentIntegration` schema.
- The live Wix site now has three additional collections (`paymentIntegrations`, `paymentRecords`, `webhookEvents`) â€” additive, no existing collection or data touched.
- Real payment collection is not yet usable end-to-end: Manor's Cremation's Clover integration is `isEnabled: false` pending real sandbox credentials (see the phase report's "Required production activation steps").
- Concurrent duplicate checkout attempts and concurrent/repeated webhook deliveries are both now guarded by real, atomic, Wix-enforced uniqueness constraints, not application-level checks â€” verified empirically against the live collections (409 responses with codes `WDE0123`/`WDE0074`) and by concurrency tests (`Promise.all` firing genuinely simultaneous requests).

## Alternatives Considered

- **Clover's legacy Developer Pay API**: explicitly excluded by this phase's own instructions â€” it is the older, more implementation-heavy integration path Clover itself is moving away from in favor of Hosted Checkout/Ecommerce.
- **An embedded Clover iframe/SDK now, instead of a redirect**: rejected for this phase as more integration surface than "launch Clover directly from a case... complete a payment on Clover's hosted page" requires; documented above as the natural next step if wanted later.
- **Storing a `PaymentRecord.checkoutUrl` nowhere and re-deriving idempotency purely from `providerCheckoutId`**: rejected â€” without a stored URL there is nothing to hand back to a caller on a duplicate request without either calling Clover again (defeating the purpose of idempotency) or leaving the caller without a checkout URL at all.
- **A dedicated "get checkout session status" call to Clover for the return-page poll**: Clover does not document such an endpoint for a Hosted Checkout session before a payment exists against it; `getPaymentStatus` instead falls back to the standard, well-documented `GET /v3/merchants/{id}/payments/{id}` resource once a `providerPaymentId` is already known from a prior webhook â€” an honest reflection of a real, confirmed gap in Clover's public API surface rather than a guessed-at endpoint.
- **A genuine multi-field unique Wix index for (organizationId, idempotencyKey)**: not possible â€” confirmed empirically and from Wix's own API schema that a unique index is capped at exactly 1 field (vs. 3 for a regular index) and a collection is capped at exactly 1 unique index total. Worked around by composing the compound semantics into a single indexed field's value instead (see "Idempotency" above) â€” the practical guarantee is identical.
- **Trusting Clover to supply a stable webhook event id**: not available â€” confirmed from Clover's own documentation that the Hosted Checkout webhook payload carries only the payment's own id, not a separate delivery/event id. A deterministic fingerprint computed from the verified payload's own stable fields was used instead, documented as a designed-around gap rather than an assumed guarantee.
